# Community Homepage Brainstorm

> Stage 0 | 2026-04-22 | 下一步：design

## 想做什么、为什么

**出发点**：用户想开发社区首页，作为学习/演示项目，但要达到商业级代码质量。

**探索过程中的关键发现**：

1. **项目定位明确**：这是学习项目，但代码质量要求达到商业级别（架构清晰、可测试、可观测、可扩展、文档完备）
2. **社区类型定位**：混合型社区（知乎 + 贴吧 + 微博的特点都有），不是单一类型
3. **架构演进策略**：渐进式微服务，先让首页跑起来，后续按需拆分服务
4. **范围收敛**：从模糊的"首页"到明确的功能清单（内容流、发布、互动、分类标签、搜索、推荐入口）

## 考虑过的方向

### 方向 A：最小首页（只浏览内容流）
- 描述：帖子列表、分页、简单排序
- 价值：最快跑起来，学习基础架构
- 代价：功能太单薄，无法展示完整能力
- 结论：**否决**（范围太小，无法达到学习目标）

### 方向 B：浏览 + 发布（核心闭环）
- 描述：帖子列表 + 发布帖子 + 查看详情
- 价值：能形成"看 → 发 → 看"闭环，架构有代表性
- 代价：复杂度适中，能做好代码质量
- 结论：**备选**（可作为起点）

### 方向 C：浏览 + 发布 + 互动
- 描述：帖子列表 + 发布 + 详情 + 点赞/评论
- 价值：像个社区了，涉及 CRUD + 关联查询
- 代价：复杂度增加，需要更多设计
- 结论：**备选**（可作为扩展）

### 方向 D：完整首页
- 描述：以上全部 + 分类/标签 + 搜索 + 推荐入口
- 价值：功能完整，能展示商业级代码能力
- 代价：复杂度最高，需要仔细设计架构
- 结论：**选定**（用户明确要求，愿意投入）

## 选定方向与遗留问题

**选定方向**：
开发完整社区首页，包含：
- 内容流（帖子列表、分页、排序）
- 发布（发帖、选择分类/标签）
- 详情（帖子详情页、作者信息）
- 互动（点赞、评论）
- 分类/标签（筛选、热门标签）
- 搜索（关键词搜索）
- 推荐入口（推荐帖子区域）

**核心行为**：用户浏览内容 → 发布内容 → 互动 → 通过分类/搜索发现更多内容


**最大未知**：
- 服务拆分边界：首页逻辑放在哪个服务？
- 数据模型设计：帖子、评论、分类、标签的表结构
- 搜索方案：MySQL LIKE vs Elasticsearch选择es
- 推荐算法：简化版用什么策略，要真实的像抖音那样得
