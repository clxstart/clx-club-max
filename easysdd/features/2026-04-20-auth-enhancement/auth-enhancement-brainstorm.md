# 认证流程增强 Brainstorm

> Stage 0 | 2026-04-20 | 下一步：design

## 想做什么、为什么

用户想要建设"超大型社区平台"（类似知乎/贴吧/微博）的完整用户体系。经过发散，确定采用分层推进策略，每轮聚焦一个模块。

第一轮聚焦**认证流程**，原因是：
- 认证是用户进入系统的入口，必须优先稳固
- 当前项目已有基础注册登录，需要补充验证机制和找回密码

## 考虑过的方向

### 方向 A：分层推进（选定）
- 每轮聚焦一个模块，分轮次验收
- 第1轮：认证流程 → 第2轮：权限系统 → 第3轮：用户资料 → 第4轮：社交关系
- **价值**：风险可控，边做边用
- **结论**：选定

### 方向 B：MVP 整体骨架（否决）
- 四个模块一次性做最小可用版本
- **否决原因**：单轮工作量大，验收复杂，容易做半截

### 方向 C：认证深度优先（否决）
- 只做认证，但做得深（多渠道、多重验证、安全加固）
- **否决原因**：用户体系其他模块暂时空白，用户只能登录不能用

### 方向 D：反向约束（部分采纳）
- 不做第三方登录、不做真实手机验证码对接
- **采纳部分**：明确边界，防止做一半发现依赖外部服务卡住

## 选定方向与遗留问题

**第一轮认证流程范围：**

| 要做 | 不做 |
|------|------|
| 找回密码（邮箱重置） | 第三方登录 |
| 图形验证码 | 真实手机验证码对接（短信服务商） |
| 邮箱验证（确保邮箱有效） | 复杂校验逻辑 |
| 手机验证码（模拟版） | 安全加固策略（登录锁定、异地提醒） |

**核心用户行为**：用户能通过邮箱/手机完成注册验证，忘记密码能自助重置。

**最大未知**：邮箱服务配置（SMTP服务器、发件人配置）。

**遗留给 design 的问题**：
1. 验证码存储方案：Redis？有效期多久？
2. 邮箱验证流程：注册时强制验证还是可选？
3. 模拟手机验证码：开发环境如何模拟？
4. 图形验证码：自建还是用第三方服务？
