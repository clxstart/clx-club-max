# 实时消息与通知系统 Brainstorm

> Stage 0 | 2026-04-25 | 下一步：design

## 想做什么、为什么

CLX 社区目前是"内容平台"形态，用户发帖后缺乏回访动力。需要实时消息与通知系统提升用户粘性和社交属性：

1. **用户粘性**——通知把用户"拉回来"（评论回复、@提醒、点赞关注）
2. **社交互动**——私信让用户之间能实时交流
3. **运营触达**——系统公告、活动推送、强制通知

探索过程中明确了**分层实时性**：不同消息类型对实时性要求不同，采用不同技术方案降低复杂度。

## 考虑过的方向

### 方向 A：实时消息与通知系统（选定）
- 描述：私信 WebSocket 实时推送 + 通知分层处理 + 在线状态显示
- 价值：从"内容平台"升级为"社交平台"，用户粘性提升
- 代价：WebSocket 长连接管理、离线消息存储、未读数一致性
- 结论：**选定**

### 方向 B：分布式积分与事务系统
- 描述：用户行为积分、跨服务事务一致性
- 价值：用户激励体系 + 分布式事务实践
- 代价：Saga/本地消息表、补偿机制、对账
- 结论：暂缓，后续 feature

### 方向 C：内容审核与风控平台
- 描述：AI 审核 + 人工复审 + 举报申诉
- 价值：合规刚需
- 代价：外部 API、工作流引擎、规则引擎
- 结论：暂缓，后续 feature

### 方向 D：个性化推荐系统
- 描述：协同过滤 + 内容特征 + 实时画像
- 价值：DAU/留存核心杠杆
- 代价：行为埋点、特征工程、向量检索
- 结论：暂缓，后续 feature

## 选定方向与遗留问题

**核心范围**：

| 模块 | 实时性 | 投递方式 | 存储 |
|------|--------|----------|------|
| 私信 | 秒级实时 | WebSocket，在线推/离线存/上线补 | 最近 N 条，过期清理 |
| 评论回复通知 | 准实时 | 轮询或延迟推送 | 持久化 |
| 点赞/关注通知 | 不追求 | 刷新时拉取 | 持久化（可聚合） |
| 系统公告 | 不追求 | 启动时检查 | 持久化 |
| 在线状态 | - | "最后活跃时间"显示 | 内存 + 定期持久化 |

**明确不做**：

- 群聊（单聊优先）
- 消息搜索（暂不引入 ES）
- 消息撤回/编辑（后续扩展）
- 多设备同步（单设备假设）
- 富媒体消息（文字优先，图片后续）

**遗留给 design 的问题**：

1. WebSocket 集群方案：单机简单，多实例怎么广播？（Redis Pub/Sub 还是专门的消息 broker）
2. 离线消息堆积：N 条上限是多少？满了怎么处理？
3. 未读数一致性：多端同时读怎么保证计数正确？
4. 心跳与断线重连：客户端/服务端怎么处理网络抖动？
5. 新服务归属：单独建 clx-message 还是放在 clx-user？
