# 点赞异步化 Brainstorm

> Stage 0 | 2026-04-24 | 下一步：design

## 想做什么、为什么

出发点是想在项目里引入更多技术栈（MQ 高级功能、Sentinel 等），既是学习目的，也是为生产做准备。经过发散讨论，最终收敛到**点赞异步化**作为切入点：

- 点赞是高频操作，同步写库在高并发下会成为瓶颈
- 点赞失败只影响计数，不丢失用户核心内容，容错空间大
- 能练到 MQ 核心技能：生产者/消费者、幂等、最终一致性

讨论中确认了其他候选（发帖、评论）但都跳过了：发帖异步化后用户看不到帖子会焦虑，评论中等复杂度但不如点赞简洁。

## 考虑过的方向

### 方向 A：纯 MQ 异步
- 描述：点赞请求发 MQ，消费后更新 DB 计数
- 价值：最简单
- 代价：用户点赞后数字不变，体验差
- 结论：否决，体验不可接受

### 方向 B：前端乐观更新 + MQ 异步
- 描述：前端立刻更新 UI，后端 MQ 异步处理
- 价值：体验好、后端简单
- 代价：前端要处理失败回滚逻辑
- 结论：备选，但想练后端一致性处理

### 方向 C：Redis 实时计数 + MQ 异步落库 + 定时校准
- 描述：点赞时 Redis 计数 + 发 MQ，消费者更新 DB，定时任务校准 Redis/DB
- 价值：体验好、后端可控、能练到一致性处理
- 代价：复杂度较高，需要处理多种边界情况
- 结论：**选定**

## 选定方向与遗留问题

**核心行为**：
1. 用户点赞 → Redis INCR + MQ 发消息 → 立刻返回成功
2. MQ 消费者 → 更新 DB `post.like_count` + 写 `like_record`
3. 定时任务 → 对比 Redis/DB 计数，修正偏差

**明显不做**：
- 本次只做点赞，不发帖/评论异步化
- 不引入 Sentinel（后续独立 feature）
- 不做分布式事务（Seata）

**最大未知**：
- 消费者失败如何重试？重试次数上限？
- 定时校准的频率？全量还是增量？
- 取消点赞（unlike）是否也异步化？

**遗留给 design 的问题**：
- MQ 消息格式设计（包含哪些字段）
- 幂等处理：重复消息如何去重
- Redis key 设计：`post:like:{postId}` vs `post:like_count:{postId}`
- 消费者失败策略：死信队列？告警？
- 定时校准任务：每分钟？每小时？全量扫描？
